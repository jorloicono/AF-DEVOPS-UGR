# Laboratorio 15: Deployments, Escalado y Actualización de Aplicaciones en Kubernetes

> **Objetivo:** Dominar los Deployments de Kubernetes para gestionar el ciclo de vida completo de una aplicación: despliegue, escalado horizontal, actualizaciones sin tiempo de inactividad (rolling updates) y rollbacks.

> **Requisitos previos:**
> - Docker Desktop con Kubernetes habilitado
> - `kubectl` instalado
> - Imagen `java-config-app:1.0` construida (ver Laboratorio 14), o construirla aquí

---

## 1️⃣ Crear dos versiones de la aplicación

Crea la carpeta del proyecto:

```powershell
mkdir k8s-deploy-lab
cd k8s-deploy-lab
```

Crea `App.java` (la versión 1.0):

```java
import com.sun.net.httpserver.HttpServer;
import java.net.InetSocketAddress;

public class App {
    static final String VERSION = "1.0";

    public static void main(String[] args) throws Exception {
        String pod = System.getenv().getOrDefault("HOSTNAME", "desconocido");
        int port = Integer.parseInt(System.getenv().getOrDefault("APP_PORT", "8080"));

        HttpServer server = HttpServer.create(new InetSocketAddress(port), 0);

        server.createContext("/", exchange -> {
            String body = String.format("Hola desde Pod: %s | Version: %s", pod, VERSION);
            byte[] r = body.getBytes();
            exchange.sendResponseHeaders(200, r.length);
            exchange.getResponseBody().write(r);
            exchange.getResponseBody().close();
        });

        server.createContext("/health", exchange -> {
            byte[] r = "OK".getBytes();
            exchange.sendResponseHeaders(200, r.length);
            exchange.getResponseBody().write(r);
            exchange.getResponseBody().close();
        });

        server.start();
        System.out.printf("v%s ejecutándose en %s:%d%n", VERSION, pod, port);
    }
}
```

Crea el `Dockerfile`:

```dockerfile
FROM eclipse-temurin:17-jdk-alpine AS build
WORKDIR /app
COPY App.java .
RUN javac App.java

FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/App.class .
EXPOSE 8080
CMD ["java", "App"]
```

Construye la versión 1.0:

```powershell
docker build -t java-deploy-app:1.0 .
```

Ahora modifica `App.java` cambiando la versión:

```java
    static final String VERSION = "2.0";
```

Construye la versión 2.0:

```powershell
docker build -t java-deploy-app:2.0 .
```

## 2️⃣ Crear un Deployment

Crea `deployment.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-app
  labels:
    app: java-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: java-app
  template:
    metadata:
      labels:
        app: java-app
    spec:
      containers:
        - name: java-app
          image: java-deploy-app:1.0
          imagePullPolicy: Never
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
```

```powershell
kubectl apply -f deployment.yaml
```

Verifica el despliegue:

```powershell
kubectl get deployment java-app
kubectl get replicaset
kubectl get pods -l app=java-app
```

Observa los componentes creados:

| Recurso | Descripción |
|---------|-------------|
| Deployment | Controlador de alto nivel que gestiona ReplicaSets |
| ReplicaSet | Garantiza el número de réplicas deseado |
| Pods | Las instancias de la aplicación (3 réplicas) |

## 3️⃣ Exponer el Deployment con un Service

Crea `service.yaml`:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: java-app-svc
spec:
  type: NodePort
  selector:
    app: java-app
  ports:
    - port: 80
      targetPort: 8080
      nodePort: 30080
```

```powershell
kubectl apply -f service.yaml
```

Prueba que el balanceo funciona ejecutando varias veces:

```powershell
for ($i = 1; $i -le 6; $i++) { Invoke-RestMethod http://localhost:30080 }
```

Observa cómo diferentes Pods responden a cada petición.

## 4️⃣ Escalado manual

### Escalar hacia arriba

```powershell
kubectl scale deployment java-app --replicas=5
kubectl get pods -l app=java-app -w
```

Espera a que todos estén en estado `Running`. Detén con `Ctrl+C`.

### Escalar hacia abajo

```powershell
kubectl scale deployment java-app --replicas=2
kubectl get pods -l app=java-app
```

Kubernetes elimina los Pods adicionales de manera controlada.

### Escalar con `kubectl edit`

```powershell
kubectl edit deployment java-app
```

Cambia el valor de `replicas` a `3` y guarda.

## 5️⃣ Rolling Update (actualización sin downtime)

Observa la actualización en una terminal mientras la ejecutas en otra:

**Terminal 1** — Observa los Pods en tiempo real:

```powershell
kubectl get pods -l app=java-app -w
```

**Terminal 2** — Actualiza la imagen a v2.0:

```powershell
kubectl set image deployment/java-app java-app=java-deploy-app:2.0
```

En la Terminal 1 verás cómo Kubernetes crea Pods nuevos y termina los antiguos progresivamente.

Verifica la actualización:

```powershell
kubectl rollout status deployment/java-app
```

Confirma que es la versión 2.0:

```powershell
for ($i = 1; $i -le 4; $i++) { Invoke-RestMethod http://localhost:30080 }
```

Revisa el historial de revisiones:

```powershell
kubectl rollout history deployment/java-app
```

## 6️⃣ Configurar la estrategia de actualización

Crea `deployment-strategy.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-app
  labels:
    app: java-app
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  selector:
    matchLabels:
      app: java-app
  template:
    metadata:
      labels:
        app: java-app
    spec:
      containers:
        - name: java-app
          image: java-deploy-app:2.0
          imagePullPolicy: Never
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 5
```

| Parámetro | Significado |
|-----------|-------------|
| `maxSurge: 1` | Máximo 1 Pod adicional durante la actualización |
| `maxUnavailable: 1` | Máximo 1 Pod no disponible durante la actualización |

```powershell
kubectl apply -f deployment-strategy.yaml
```

## 7️⃣ Rollback a una versión anterior

Regresa a la versión 1.0:

```powershell
kubectl rollout undo deployment/java-app
kubectl rollout status deployment/java-app
```

Verifica:

```powershell
Invoke-RestMethod http://localhost:30080
```

### Rollback a una revisión específica

```powershell
kubectl rollout history deployment/java-app
kubectl rollout undo deployment/java-app --to-revision=2
```

---

## 8️⃣ Pausar y reanudar actualizaciones

Útil para actualizar múltiples propiedades antes de desplegar:

```powershell
kubectl rollout pause deployment/java-app
```

Ahora haz cambios sin que se desplieguen:

```powershell
kubectl set image deployment/java-app java-app=java-deploy-app:2.0
kubectl set resources deployment/java-app -c java-app --limits=memory=256Mi,cpu=200m
```

Los cambios NO se aplican hasta reanudar:

```powershell
kubectl rollout resume deployment/java-app
kubectl rollout status deployment/java-app
```

```powershell
kubectl delete deployment java-app
kubectl delete service java-app-svc

```

## ✅ Resumen

En este laboratorio has aprendido a:

- Crear **Deployments** que gestionan réplicas de Pods automáticamente
- **Escalar** horizontalmente una aplicación (`scale --replicas`)
- Realizar **rolling updates** para actualizar sin interrupción de servicio
- Configurar la **estrategia de actualización** (`maxSurge`, `maxUnavailable`)
- Hacer **rollback** a versiones anteriores con `rollout undo`
- **Pausar y reanudar** despliegues para agrupar cambios
- Usar **readinessProbe** para asegurar actualizaciones seguras
