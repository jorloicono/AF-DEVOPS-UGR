# Laboratorio 16: Introducción a Ansible y su Arquitectura

> **Objetivo:** Comprender la arquitectura de Ansible, instalar el entorno de trabajo usando Docker (ya que Ansible no corre nativamente en Windows), ejecutar comandos ad-hoc y entender los componentes: inventario, módulos, hechos (facts) y la conexión SSH.

> **Requisitos previos:**
> - Docker Desktop instalado y en ejecución
> - Terminal PowerShell

---

## 1️⃣ Entender la arquitectura de Ansible

Ansible sigue un modelo **sin agentes** (agentless):

| Componente | Descripción |
|------------|-------------|
| **Nodo de control** | Máquina donde se ejecuta Ansible (Linux/macOS) |
| **Nodos gestionados** | Servidores destino accesibles por SSH |
| **Inventario** | Archivo que lista los nodos gestionados |
| **Módulos** | Unidades de trabajo (instalar paquete, copiar archivo, etc.) |
| **Playbooks** | Archivos YAML que definen tareas a ejecutar |
| **Facts** | Información recopilada automáticamente de los nodos |

> **Nota sobre Windows:** Ansible requiere Linux como nodo de control. Usaremos contenedores Docker para simular el entorno completo.

## 2️⃣ Crear el entorno de laboratorio con Docker

Crea la carpeta del proyecto:

```powershell
mkdir ansible-intro-lab
cd ansible-intro-lab
```

### Nodo gestionado (servidor con SSH y Java)

Crea `Dockerfile.node`:

```dockerfile
FROM eclipse-temurin:17-jdk-alpine

RUN apk add --no-cache openssh python3 bash && \
    ssh-keygen -A && \
    adduser -D devops && \
    echo "devops:devops123" | chpasswd && \
    sed -i 's/#PermitRootLogin.*/PermitRootLogin no/' /etc/ssh/sshd_config && \
    sed -i 's/#PasswordAuthentication.*/PasswordAuthentication yes/' /etc/ssh/sshd_config

EXPOSE 22
CMD ["/usr/sbin/sshd", "-D"]
```

### Nodo de control (Ansible)

Crea `Dockerfile.control`:

```dockerfile
FROM python:3.11-alpine

RUN apk add --no-cache openssh-client sshpass bash && \
    pip install --no-cache-dir ansible

RUN mkdir -p /root/.ssh && \
    ssh-keygen -t rsa -b 2048 -f /root/.ssh/id_rsa -N "" && \
    echo "StrictHostKeyChecking no" > /root/.ssh/config

WORKDIR /ansible
CMD ["sleep", "infinity"]
```

### Compose para levantar ambos

Crea `docker-compose.yml`:

```yaml
services:
  control:
    build:
      context: .
      dockerfile: Dockerfile.control
    container_name: ansible-control
    hostname: control
    volumes:
      - ./ansible:/ansible
    networks:
      - ansible-net
    depends_on:
      - node1
      - node2

  node1:
    build:
      context: .
      dockerfile: Dockerfile.node
    container_name: ansible-node1
    hostname: node1
    networks:
      - ansible-net

  node2:
    build:
      context: .
      dockerfile: Dockerfile.node
    container_name: ansible-node2
    hostname: node2
    networks:
      - ansible-net

networks:
  ansible-net:
    driver: bridge
```

Crea la carpeta de trabajo de Ansible:

```powershell
mkdir ansible
```

Construye y arranca el entorno:

```powershell
docker compose up -d --build
```

Verifica que los tres contenedores están corriendo:

```powershell
docker compose ps
```

## 3️⃣ Crear el inventario

Crea el archivo `ansible/inventory.ini`:

```ini
[servidores]
node1 ansible_host=node1
node2 ansible_host=node2

[servidores:vars]
ansible_user=devops
ansible_password=devops123
ansible_connection=ssh
```

Prueba la conectividad con el módulo `ping`:

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m ping
```

Deberías ver `SUCCESS` para ambos nodos.

## 4️⃣ Comandos ad-hoc

Los comandos ad-hoc ejecutan un módulo en uno o más nodos sin necesidad de un Playbook.

### Ejecutar un comando en todos los nodos

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "hostname"
```

### Ver información del sistema

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "uname -a"
```

### Verificar que Java está instalado

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "java -version"
```

### Usar el módulo `shell` para comandos complejos

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m shell -a "echo Hola desde $(hostname) && date"
```

### Copiar un archivo a los nodos

Primero, crea un archivo de prueba:

```powershell
"Archivo de prueba desde Ansible" | Set-Content ansible/prueba.txt
```

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m copy -a "src=/ansible/prueba.txt dest=/home/devops/prueba.txt"
```

Verifica:

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "cat /home/devops/prueba.txt"
```

### Crear un directorio

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m file -a "path=/home/devops/mi-app state=directory owner=devops"
```

## 5️⃣ Recopilar Facts (información del sistema)

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini node1 -m setup
```

Esto muestra TODA la información que Ansible recopila. Para filtrar:

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini node1 -m setup -a "filter=ansible_os_family"
docker exec ansible-control ansible -i /ansible/inventory.ini node1 -m setup -a "filter=ansible_hostname"
docker exec ansible-control ansible -i /ansible/inventory.ini node1 -m setup -a "filter=ansible_default_ipv4"
```

| Fact clave | Ejemplo de valor |
|------------|-----------------|
| `ansible_hostname` | node1 |
| `ansible_os_family` | Alpine |
| `ansible_distribution` | Alpine |
| `ansible_default_ipv4.address` | 172.18.0.X |

## 6️⃣ Módulos más usados

Prueba estos módulos ad-hoc para entender lo que Ansible puede hacer:

### Módulo `user` — Crear un usuario

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m user -a "name=appuser state=present" --become -K
```

> **Nota:** `--become` ejecuta con privilegios elevados (como sudo). Si pide contraseña usa `-K` o añade `ansible_become_password=devops123` al inventario.

Sin contraseña sudo, usa directamente:

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "id appuser" --become
```

### Módulo `package` — Instalar un paquete

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "apk add curl" --become
```

Verifica:

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "curl --version"
```

**Módulos comunes de Ansible:**

| Módulo | Uso |
|--------|-----|
| `ping` | Verificar conectividad |
| `command` | Ejecutar un comando simple |
| `shell` | Ejecutar comandos con pipes/redirección |
| `copy` | Copiar archivos al nodo |
| `file` | Gestionar archivos y directorios |
| `user` | Gestionar usuarios |
| `service` | Gestionar servicios del sistema |
| `template` | Copiar archivos con variables Jinja2 |
| `debug` | Mostrar mensajes de depuración |

## 7️⃣ Configuración de Ansible (ansible.cfg)

Crea `ansible/ansible.cfg` para evitar repetir opciones:

```ini
[defaults]
inventory = /ansible/inventory.ini
host_key_checking = False
remote_user = devops

[privilege_escalation]
become = False
```

Ahora los comandos son más cortos:

```powershell
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m ping
docker exec ansible-control ansible -i /ansible/inventory.ini servidores -m command -a "java -version"
```
